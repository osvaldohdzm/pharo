| **BUSINESS LOGIC FLAW (CMS)** | Testing revealed that a Business Logic Flaw existed within the assessed application. Attacks on the business logic of an application can have significant impacts, furthermore, are difficult to identify as they are specific to the application being tested and require an understanding of that application - manual testing is typically required to highlight these. Business logic flaws may affect business rules, policies or workflows based on ordered tasks from one participant or stage to another. The Business Logic Flaw identified is that an unauthorized client (attacker) can be able to upload files / contracts of any extension for different users, after that it is possible to visualize the information without required any token (without authentication). | This can lead to a wide range of problems like information exposure. An attacker could upload and read sensitive data of the different clients requesting the URLs of the web application without an authentication token. | Modify the application's logic so that it maps (and enforces) business rules, policies and/or workflows throughout. For web applications, make sure that the access control mechanism is enforced correctly at the server side on every page. Users should not be able to access any unauthorized functionality or information by simply requesting direct access to that page. A review should be undertaken to determine which file types are expected to be received. Only these files should be allowed to be uploaded, any not on an explicitly accepted list must be blocked at source. |

| **BROKEN AUTHENTICATION (CMS)** | It was found that the web application doesn’t validate properly the session identifiers prior to process requests, this allows an attacker to interact with the application without supplying credentials (an access control token). In other words, it is possible to bypass the authentication mechanism by directly calling an internal page that should only be available to a user once they have authenticated. | A malicious user could exploit this vulnerability to access sensitive information and/or application functionality without authenticating. | Validate properly the session identifiers (token) prior to process requests. It must be ensured that access controls are deployed uniformly throughout the application. Preventing unauthorized URL access requires selecting an approach for requiring proper authentication and proper authorization for each page of all module or options of the application. |
| **PRIVILEGE ESCALATION (CMS)** | Privilege escalation occurs when a user gets access to more resources or functionality than they are normally allowed, and such elevation or changes should have been prevented by the application. It was identified that is possible to upload files for any other user, knowing that the ID of user. | The result is that the application performs actions with more privileges than those intended by the developer or system administrator. | Application must ensure server-side validation and access controls to prevent client-side manipulation. Also, entitlements on server-side must ensure the client session is allowed to the object being requested or action being performed. User privileges and application features should be linked in RBAC models whenever possible. |
| **ABUSE OF OTP FUNCTIONALITY (CMS)** | The application allows users to manipulate the recipient email address during the OTP (One-Time Password) delivery process. As a result, it enables an attacker to redirect the OTP to an arbitrary email address just changing the identification number of a user without authentication. | An attacker can redirect the OTP emails to other users of the web application and know emails of other clients. | All client-side input should be validated before being processed by the application using data validation techniques. The backend must strictly enforce OTP delivery only to the email address associated with the authenticated user or the account being verified. Use an authorization mechanism to check if the logged-in user has access to perform the requested action. |
| **TWO FACTOR AUTHENTICATION BYPASS (CMS)** | An attacker with valid credentials (e.g. obtained through a phishing or brute force attack), can drop the verification OTP request through a proxy and log in successfully, completely bypassing the second factor authentication. The authentication token (Fepweb-Jwt-Token) is issued before the OTP is validated. | The bypass allows attackers with valid credentials (e.g. from phishing or credential stuffing) to gain full access to accounts without needing the second factor, defeating the purpose of 2FA entirely. | Implement strict server-side controls that enforce 2FA completion before granting access to sensitive areas or actions. Consider using separate token states (e.g. auth_status: “2fa_pending” vs. “authenticated”) and enforce checks on the backend. |
