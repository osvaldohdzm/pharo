| BUSINESS LOGIC FLAW (CMS) | Testing revealed that a Business Logic Flaw existed within the assessed application. Attacks on the business logic of an application can have significant impacts, furthermore, are difficult to identify as they are specific to the application being tested and require an understanding of that application - manual testing is typically required to highlight these. Business logic flaws may affect business rules, policies or workflows based on ordered tasks from one participant or stage to another. The Business Logic Flaw identified is that an unauthorized client (attacker) can be able to upload files / contracts of any extension for different users, after that it is possible to visualize the information without required any token (without authentication). 

 | This can lead to a wide range of problems like information exposure. An attacker could upload and read sensitive data of the different clients requesting the URLs of the web application without an authentication token. 

 | Modify the application's logic so that it maps (and enforces) business rules, policies and/or workflows throughout. For web applications, make sure that the access control mechanism is enforced correctly at the server side on every page. Users should not be able to access any unauthorized functionality or information by simply requesting direct access to that page. A review should be undertaken to determine which file types are expected to be received. Only these files should be allowed to be uploaded, any not on an explicitly accepted list must be blocked at source.

| BROKEN AUTHENTICATION (CMS) | It was found that the web application doesn’t validate properly the session identifiers prior to process requests, this allows an attacker to interact with the application without supplying credentials (an access control token). In other words, it is possible to bypass the authentication mechanism by directly calling an internal page that should only be available to a user once they have authenticated. 

 | A malicious user could exploit this vulnerability to access sensitive information and/or application functionality without authenticating. 

 | Validate properly the session identifiers (token) prior to process requests. It must be ensured that access controls are deployed uniformly throughout the application. Preventing unauthorized URL access requires selecting an approach for requiring proper authentication and proper authorization for each page of all module or options of the application. Frequently, such protection is provided by one or more components external to the application code. Regardless of the mechanism(s), it is recommended that: 1. The authentication and authorization policies be role based, to minimize the effort required to maintain these policies.  2. The policies should be highly configurable, in order to minimize any hard coded aspects of the policy.  3. The enforcement mechanism(s) should deny all access by default, requiring explicit grants to specific users and roles for access to every page.  4. If the page is involved in a workflow, check to make sure the conditions are in the proper state to allow access.

| PRIVILEGE ESCALATION (CMS) | Privilege escalation occurs when a user gets access to more resources or functionality than they are normally allowed, and such elevation or changes should have been prevented by the application. This is usually caused by a flaw in the application. Usually, people refer to vertical escalation when it is possible to access resources granted to more privileged accounts (e.g., acquiring administrative privileges for the application), and to horizontal escalation when it is possible to access resources granted to a similarly configured account (e.g., in an online banking application, accessing information related to a different user). 

 | The result is that the application performs actions with more privileges than those intended by the developer or system administrator. 

 | Application must ensure server-side validation and access controls to prevent client-side manipulation. Also, entitlements on server-side must ensure the client session is allowed to the object being requested or action being performed. User privileges and application features should be linked in RBAC models whenever possible to guarantee distinctive access profiles and easy customization.

The application allows users to manipulate the recipient email address during the OTP (One-Time Password) delivery process. As a result, it enables an attacker to redirect the OTP to an arbitrary email address just changing the identification number of a user without authentication. 

 | An attacker can redirect the OTP emails to other users of the web application and know emails of other clients. | * All client-side input should be validated before being processed by the application using data validation techniques. * The backend must strictly enforce OTP delivery only to the email address associated with the authenticated user or the account being verified. * Use an authorization mechanism to check if the logged-in user has access to perform the requested action.  * Validate user stories if a request requires a token, it should be validated for one operation and matched with user identifier each time that the user attempts to perform an operation.

| TWO FACTOR AUTHENTICATION BYPASS (CMS) | An attacker with valid credentials (e.g. obtained through a phishing or brute force attack), can drop the verification OTP request through a proxy and log in successfully, completely bypassing the second factor authentication. 

 | The bypass allows attackers with valid credentials (e.g. from phishing or credential stuffing) to gain full access to accounts without needing the second factor, defeating the purpose of 2FA entirely. 

 | Implement strict server-side controls that enforce 2FA completion before granting access to sensitive areas or actions. Consider using separate token states (e.g. auth_status: “2fa_pending” vs. “authenticated”) and enforce checks on the backend.

 | A best practice security measure is to ensure users are restricted to a single logon. Unless a valid business justification exists, multiple logons should not be allowed. | PLATFORM INFORMATION DISCLOSURE (robots.txt) | The web server contains a robots.txt file, however the file is empty. The file robots.txt is used to give instructions to web robots, such as search engine crawlers, about locations within the web site that robots are allowed, or not allowed, to crawl and index. 

 | The presence of the robots.txt does not in itself present any kind of security vulnerability. However, it is often used to identify restricted or private areas of a site's contents. The information in the file may therefore help an attacker to map out the site's contents, especially if some of the locations identified are not linked from elsewhere in the site. If the application relies on robots.txt to protect access to these areas, and does not enforce proper access control over them, then this presents a serious vulnerability. 

 | The robots.txt file is not itself a security threat, and its correct use can represent good practice for non-security reasons. You should not assume that all web robots will honor the file's instructions. Rather, assume that attackers will pay close attention to any locations identified in the file. Do not rely on robots.txt to provide any kind of protection over unauthorized access.
